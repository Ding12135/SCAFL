SCAFL 工程代码快速阅读指南（面向实验准备）

适用目标
- 你希望在最短时间内理解工程全貌，并能独立修改配置、跑实验、解释结果。
- 你不追求一次性吃透全部细节，而是先建立“流程主线 + 关键变量 + 日志映射”。

------------------------------------------------------------
一、推荐总时长与产出
------------------------------------------------------------

建议总时长：2~4 小时（分两轮）

第一轮（建立地图，约 1.5~2 小时）
- 目标：知道系统怎么跑、关键决策在哪、日志怎么看。
- 产出：
  1) 一张事件流程图（从客户端更新到服务器聚合再到状态更新）。
  2) 一张变量流表（V, Q_k, tau_k, tau_max, D_t, beta_k, objective_p2）。

第二轮（面向实验，约 1~2 小时）
- 目标：知道哪些配置会影响哪些指标，实验改动不会“盲调”。
- 产出：
  1) “配置项 -> 指标变化”对照表。
  2) “最小可复现实验”操作清单（参数、日志、随机种子）。

------------------------------------------------------------
二、阅读顺序（强烈建议按顺序）
------------------------------------------------------------

Step 0：先看文档（10 分钟）
1) docs/implementation_status.md
   - 先明确：当前是“论文启发实现”，不是严格复现。
2) docs/paper_symbol_mapping.md
   - 先建立符号映射，避免后续看到变量名时反复猜测。

Step 1：主流程入口（优先级最高）
文件：afl/server.py
阅读目标：
- 搞清一轮逻辑（或一次 aggregation event）的完整生命周期：
  收到更新 -> 构建候选 -> 策略选择 -> 聚合 -> 更新 runtime state / queue -> 写日志。
- 确认策略函数在什么位置被调用，调用时带了哪些输入。
- 确认哪些地方更新了 per-client 状态和全局统计量。

你要记录的问题：
- 决策触发条件是什么？（周期、事件、阈值）
- 聚合触发条件是什么？
- “round / event / step” 三个时间概念如何对应？
- 写了哪些 CSV、每个 CSV 在什么时刻写？

Step 2：策略核心（第二优先级）
文件：afl/scafl_policy.py
阅读目标：
- 看懂 objective_p2 的计算分解（time term + queue-related term）。
- 看懂 prefix 枚举与选优过程（只在排序前缀上搜索）。
- 看懂 V, tau_max, beta 的来源与作用范围。
- 看懂诊断日志字段如何从中间变量生成。

你要记录的问题：
- 输入 candidate 的结构是什么？
- D_t 如何计算（max d_k）？d_k 如何构成？
- beta 是 client-level 还是 update-level？
- tau_k 在多 pending 时如何聚合？
- 返回结果除了“选中子集”外，还有哪些调试信息？

Step 3：候选构建与聚合语义
文件：afl/aggregator.py
阅读目标：
- 理解 pending updates 如何维护、过滤和快照化。
- 理解 staleness、compute_time、upload_delay 等字段的来源。
- 理解“聚合后的模型更新”与“状态清理”顺序。

你要记录的问题：
- 候选集合是按什么规则进入/移出 buffer 的？
- 是否存在去重或合并逻辑？
- 聚合后哪些字段会被重置或递增？

Step 4：状态结构与类型定义
文件：afl/runtime_state.py, afl/scafl_types.py
阅读目标：
- 明确每个 dataclass/结构体代表什么。
- 明确 virtual_queue、staleness、pending 元数据分别存在哪里。
- 明确“逻辑上每个 client 的状态”与“单条 update 的状态”区别。

你要记录的问题：
- 哪些字段是决策输入，哪些只是日志用途？
- 字段更新的唯一来源函数有哪些？

Step 5：动态控制器与参数注入
文件：afl/dynamic_controller.py
阅读目标：
- 弄清 tau_max_t 或相关动态参数如何计算、何时覆盖策略默认值。
- 明确控制器与策略的边界（谁决定上界、谁执行优化）。

Step 6：剩余支持模块（快速扫读）
文件：afl/client.py, afl/data.py, afl/model.py, afl/utils.py
阅读目标：
- client 侧训练步骤及上报字段定义；
- 数据划分与随机性来源；
- 模型结构是否影响训练时延和梯度特征；
- 通用工具函数是否对实验有隐性影响。

------------------------------------------------------------
三、按“问题驱动”阅读（每个函数都问这四件事）
------------------------------------------------------------

阅读任意函数时，只回答四个问题：
1) 输入来自哪里？（谁调用它）
2) 输出给了谁？（返回值和副作用）
3) 改了哪些状态？（尤其是 queue、pending、round 统计）
4) 对实验结果有什么影响？（精度、时延、公平性、稳定性）

如果这四个问题回答不清，就不要继续深入下一层细节。

------------------------------------------------------------
四、关键变量追踪模板（建议边读边填）
------------------------------------------------------------

建议你在笔记里建一个表，至少包含以下列：
- 变量名
- 代码定义位置
- 更新位置
- 参与决策的位置
- 对应日志字段
- 可能的论文语义偏差

重点变量清单：
- V
- Q_k / virtual_queue
- tau_k / staleness
- tau_max_t / tau_max_used
- d_k / D_t
- beta_k
- objective_p2
- candidate_term_sum

------------------------------------------------------------
五、日志反向读代码（非常省时间）
------------------------------------------------------------

你可以从日志字段倒推代码语义，推荐优先看：
1) round_metrics.csv
   - 看整体选优结果、关键目标值和参数快照。
2) p2_prefix_diagnostics.csv
   - 看每个 prefix 的 objective 变化，验证“为什么选了这个前缀”。
3) decision_debug.csv
   - 看更细粒度变量分解（q_k_pre, tau_k, beta_k, term_k）。

推荐操作方式：
- 挑一轮“异常轮次”（指标突变）作为样本；
- 从 CSV 定位该轮；
- 再回代码看该轮对应逻辑；
- 这样比从代码线性读快很多。

------------------------------------------------------------
六、实验前必须确认的 10 件事
------------------------------------------------------------

1) 随机种子是否固定，固定位置在哪。
2) 数据划分是否固定，是否可重复。
3) 客户端数量、并发、到达模式是否固定。
4) policy.V 当前值及配置来源是否正确。
5) tau_max 来源是控制器给定还是策略回退。
6) beta 语义是 client-level 还是 update-level（按当前实现确认）。
7) D_t 的计算方式是否与你的实验假设一致。
8) 关键 CSV 是否完整写出且字段无缺失。
9) 评估指标在什么时刻采样（每轮、每事件、间隔采样）。
10) 你的实验结论是否会被“工程近似”影响（写在实验记录里）。

------------------------------------------------------------
七、两轮阅读法（可直接执行）
------------------------------------------------------------

第一轮：只画图，不深挖实现细节
- 输出 A：事件时序图（server 主线）
- 输出 B：变量流图（policy 主线）
- 输出 C：日志字段到变量映射

第二轮：只解决实验会改到的点
- 你准备改哪些参数，就只追这些参数的读取、传递和生效路径。
- 你准备看哪些指标，就只追这些指标的计算和写盘路径。

避免陷阱：
- 不要第一轮就逐行读 1000+ 行文件；
- 不要在 utility 函数里过早停留；
- 不要在没有日志对应关系前做“语义猜测”。

------------------------------------------------------------
八、建议你马上执行的最小动作
------------------------------------------------------------

1) 打开 server.py，只找“策略调用”“聚合调用”“日志写入”三个锚点。
2) 打开 scafl_policy.py，只找“objective 计算”“prefix 选择”“返回诊断信息”三个锚点。
3) 用一轮真实输出 CSV，对照上述锚点做一次“单轮回放”。

完成这三步后，你基本就可以开始设计实验，而不是继续盲读代码。

------------------------------------------------------------
附：阅读完成的判定标准
------------------------------------------------------------

当你能清楚回答下面问题时，就算“读懂到可实验”：
- 为什么这一轮选了这个前缀？
- 这个选择是由哪个变量主导的（V、Q、tau、D_t）？
- 若我要改变策略行为，该改哪一层（配置、控制器、策略、候选构建）？
- 我看到的日志变化，能否回溯到明确代码路径？

如果以上四问都能回答，建议立即进入实验，而不是继续扩大阅读范围。
